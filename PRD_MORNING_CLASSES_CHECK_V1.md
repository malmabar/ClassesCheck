## 1. الملخص التنفيذي

هذا المستند يحدد متطلبات تحويل ملف Excel الحالي إلى نظام ويب متكامل داخل منصة SSP مع الحفاظ على منطق العمل والنتائج التشغيلية الحالية.

الهدف من الإصدار الأول:
1. استبدال التشغيل اليدوي داخل Excel (`تحديث` + `RUN` + `حفظ`) بتدفق نظامي موثوق.
2. الحفاظ على نفس مخرجات الأعمال الأساسية:
   - قاعات التشغيل (`Halls_Copy`)
   - جدول الشعب النهائي (`CRNs_Copy`)
   - توزيع المدربين (`TrainersSC`)
   - مؤشرات التوزيع النسبي (`التوزيع النسبي`)
3. نقل القواعد من معادلات Excel/VBA إلى محرك قواعد قابل للاختبار والصيانة.
4. تقديم واجهة عربية RTL بنفس أسلوب العرض التشغيلي قدر الإمكان.

## 2. المشكلة الحالية

### 2.1 الوضع الحالي (As-Is)
العمل اليوم يعتمد على ملف Excel كبير متعدد الشيتات ومعادلات كثيفة:

1. 14 شيت، منها شيتات تشغيلية وشيتات حسابية مخفية.
2. 97 نطاقًا مسمى (Named Ranges).
3. VBA Macros فعالة للتشغيل والاستيراد والحفظ.
4. أحجام معادلات كبيرة جدًا في شيتات حرجة:
   - `Codes`: ~134,990 معادلة
   - `CRNs`: ~160,596 معادلة
   - `Rooms`: ~21,411 معادلة
   - `TrainersSC`: ~26,108 معادلة
5. قواعد تنسيق شرطي (Conditional Formatting) معقدة خاصة في `CRNs_Copy`.

### 2.2 ألم التشغيل الحالي
1. الاعتماد على جهاز/نسخة Excel محددة.
2. هشاشة التعديلات: أي تغيير صيغة قد يكسر نتائج كاملة.
3. صعوبة تتبع من غيّر ماذا ومتى.
4. صعوبة الاختبارات الآلية والمطابقة.
5. قابلية التوسع محدودة عند زيادة البيانات أو المستخدمين.

## 3. الأهداف والنطاق

### 3.1 أهداف الإصدار (In Scope)
1. تحويل المنطق الحالي كاملًا إلى Backend Rules + Data Pipeline.
2. بناء واجهة تشغيل تماثل الشاشة الحالية (Homepage) وتدفقها.
3. دعم استيراد مصدر SS01 (CSV) بنفس منطق `get_Data_From_File`.
4. إنتاج مخرجات مماثلة لـ `Halls_Copy`, `CRNs_Copy`, `TrainersSC`, `التوزيع النسبي`.
5. دعم تصدير Excel/PDF من نفس المخرجات.
6. إنشاء سجل تشغيلي (Run History) ومراجعة الأخطاء.

### 3.2 خارج نطاق الإصدار الأول (Out of Scope)
1. بناء محرك جدولة جديد مستقل عن قواعد Excel قبل تحقيق التكافؤ.
2. إعادة تصميم أعمال المؤسسة (Business Policy Redesign).
3. تكاملات خارجية مباشرة مع ERP في هذا الإصدار.

### 3.3 مبدأ التنفيذ
1. **Parity First**: أولًا مطابقة النتائج الحالية، ثم التحسين لاحقًا.
2. **No Formula Runtime**: لا تشغيل مباشر لصيغ Excel داخل الإنتاج.
3. **Deterministic Runs**: نفس المدخلات يجب أن تعطي نفس المخرجات دائمًا.

## 5. تحليل الملف الحالي (Source Analysis)

## 5.1 تدفق العمل داخل Excel

### زر `تحديث` (Macro: `get_Data_From_File`)
1. يطلب ملف CSV من المستخدم.
2. يمسح نطاق `SS01_Report!A1:S5000`.
3. ينسخ بيانات الملف إلى `SS01_Report` كقيم.
4. يحدث طابع الوقت في `Homepage!G22`.

### زر `RUN` (Macro: `copyDataToAnotherSheet`)
1. يجهز `Rooms` ثم ينسخ الناتج إلى `Halls_Copy` كقيم.
2. يجهز `CRNs` ثم ينسخ الناتج إلى `CRNs_Copy` كقيم.
3. يحدّث طابع الوقت في `Halls_Copy`, `CRNs_Copy`, `TrainersSC`.
4. يعتمد على الحسابات المسبقة في الشيتات المخفية.

### زر `حفظ` (Macro: `copyDataToAnotherWorkbook`)
1. ينشئ ملف Excel جديد.
2. ينسخ شيت `Halls_Copy` إليه.

## 7. المتطلبات الوظيفية

## 7.1 إدارة المدخلات والاستيراد

### FR-001 استيراد SS01
1. النظام يدعم رفع CSV لـ SS01.
2. يدعم الترميز `UTF-8` (مع/بدون BOM) و`Windows-1256`.
3. يدعم الفاصل `,` أو `;` مع الاكتشاف التلقائي.
4. يتحقق من الأعمدة المطلوبة قبل التنفيذ بالاسم المطابق.
5. الأعمدة الحرجة الإلزامية: `الفصل التدريبي`, `القسم`, `المقرر`, `اسم المقرر`, `الرقم المرجعي`, `نوع الشعبة`, `اليوم`, `الوقت`, `مبنى`, `قاعة`, `سعة`, `مسجلين`, `رقم المدرب`, `اسم المدرب`.
6. يحفظ Snapshot للملف الخام مع `run_id` و`checksum`.
7. يرفض التشغيل عند غياب عمود حرج أو فشل parsing في الأعمدة الحرجة.
8. الصفوف غير الصالحة تحفظ في `mc_import_rejects` مع سبب الرفض.

### FR-002 إدارة المراجع
1. النظام يحمل بيانات `Building&Rooms`, `TrainingPlan`, `SL03-TSM`.
2. يدعم تحديثها بإصدارات (Versioned).
3. يمنع تشغيل Run بدون مرجع أساسي صالح.

## 7.2 إعدادات التشغيل

### FR-003 إعدادات الجلسة
1. `semester` إلزامي.
2. `MorOREve` إلزامي (`صباحي`/`مسائي`).
3. حفظ الإعدادات مع كل Run.

## 7.3 محرك القواعد الحسابية

### FR-004 طبقة Codes
1. اشتقاق الحقول الأساسية من SS01.
2. تصنيف صباحي/مسائي.
3. تفكيك اليوم/الوقت إلى مصفوفة حصص.
4. ربط المدرب بالقسم من مرجع SL03.

### FR-005 طبقة Rooms
1. بناء شبكة إشغال القاعات لكل غرفة.
2. الحساب يعتمد على 5 أيام و8 حصص يوميًا.
3. حساب الإشغال (`AV`) وقائمة المقررات (`AW`).
4. دعم التصفية حسب الفترة.

### FR-006 طبقة CRNs
1. تجميع بيانات الشعبة على مستوى CRN.
2. حساب الأحمال الزمنية لكل يوم/حصة.
3. حساب `س.نظام` و`س.خطة` والمقارنة بينهما.
4. إبراز حالات التعارض/النقص حسب قواعد التحقق.

### FR-007 طبقة TrainersSC
1. بناء مصفوفة أحمال المدرب الأسبوعية.
2. حساب مجموع العبء (`AU`) ومؤشرات Day-off.
3. إبراز الحالات الحدية بالألوان كما في Excel.

### FR-008 طبقة التوزيع النسبي
1. حساب نسب توزيع الأحمال على الخانات الزمنية.
2. حساب نسب يومية وأسبوعية.
3. عرض Heatmap مطابق للدلالات الحالية.

## 7.4 نشر النتائج

### FR-009 إنشاء نسخ نهائية
1. إنشاء `Halls_Copy` كنسخة قيم.
2. إنشاء `CRNs_Copy` كنسخة قيم.
3. إنشاء `TrainersSC` كنسخة منشورة مرتبطة بـ `run_id`.
4. إنشاء `التوزيع النسبي` كنسخة منشورة مرتبطة بـ `run_id`.
5. تحديث طابع وقت التشغيل على النتائج.

### FR-010 التصدير
1. تصدير Excel مطابق للأعمدة والتنسيق الأساسي.
2. تصدير PDF عرضي للنتائج الرئيسية.
3. تسمية الملفات حسب الفصل/الفترة/وقت التشغيل.

## 7.5 التشغيل والمراجعة

### FR-011 سجل التشغيلات
1. لكل Run: مدخلات، نسخة مراجع، نتائج، مدة التنفيذ، الحالة.
2. حفظ رسائل التحذير/الأخطاء.
3. دعم المقارنة بين Runين.
4. حفظ `rule_version` و`input_checksum` و`reference_version` لكل Run.

### FR-012 صلاحيات التشغيل
1. فقط دور الإدارة يستطيع: استيراد، تشغيل، نشر، تصدير.
2. أدوار القراءة يمكنها عرض النتائج فقط.

### FR-013 دورة حياة التشغيل (Run Lifecycle)
1. حالات التشغيل المعتمدة: `CREATED`, `VALIDATING`, `RUNNING`, `FAILED`, `SUCCEEDED`, `PUBLISHED`.
2. منع تشغيلين متوازيين لنفس السياق (`semester + period`) عبر lock تشغيلي.
3. دعم `idempotency_key` مبني على `input_checksum + reference_version + settings` لمنع التشغيل المكرر.
4. دعم إعادة المحاولة التلقائية مرة واحدة للأخطاء العابرة (مثل timeout)، ثم التحويل إلى `FAILED`.
5. لا ينتقل Run إلى `PUBLISHED` إلا بعد نجاح checks الخاصة بالمطابقة وقواعد التحقق.

## 8. متطلبات واجهة المستخدم (UI/UX)

## 8.1 شاشة رئيسية بديلة لـ Homepage
1. عناصر إلزامية:
   - اختيار الفصل
   - اختيار الفترة (صباحي/مسائي)
   - فلتر رئيسي للفترة (`الكل`/`صباحي`/`مسائي`) ينعكس على كل الشاشات والنتائج
   - زر `تحديث البيانات`
   - زر `تشغيل`
   - زر `حفظ/تصدير`
2. إظهار آخر وقت استيراد وآخر وقت تشغيل.
3. إظهار حالة الجاهزية قبل التشغيل.

## 8.2 شاشات النتائج
1. تبويب `Halls` (مطابق `Halls_Copy`).
2. تبويب `CRNs` (مطابق `CRNs_Copy`).
3. تبويب `Trainers` (مطابق `TrainersSC`).
4. تبويب `Distribution` (مطابق `التوزيع النسبي`).
5. دعم التجميد، التصفية، البحث، الفرز.

## 8.3 الفلاتر الشاملة (Global Filters)
1. يجب وجود فلتر فترة رئيسي في الترويسة يحدد العرض (`الكل`/`صباحي`/`مسائي`).
2. الفلتر الرئيسي يطبق على كل الصفحات/التبويبات بدون استثناء.
3. حالة الفلتر تحفظ (Persisted) وتستعاد تلقائيًا عند إعادة فتح النظام.
4. أي API جدولية مرتبطة بالعرض يجب أن تقبل filter period بشكل صريح.

## 8.4 مطابقة التلوين
1. الخلايا بقيمة `1` في مصفوفات الأيام تظهر بلون الحالة المعتمد.
2. قواعد النقص/الزيادة في `CRNs` و`Trainers` تطبق بصريًا بنفس منطق Excel.
3. الحالات الفارغة/غير الصالحة تميّز بوضوح.

## 8.5 نظام التصميم (Design System)
1. اعتماد Design Tokens موحدة للألوان والمسافات والظلال والحواف.
2. فصل الواجهة إلى طبقتين:
   - `tokens.css`
   - `components.css`
3. الالتزام بنمط `Bento Grid` للوحات التحكم والنتائج.
4. استخدام أيقونات حديثة موحدة (Lucide أو بديل محلي مكافئ) في:
   - البطاقات الإحصائية
   - الأزرار الأساسية/الثانوية
   - حالات الفراغ (Empty States)
5. أي إضافة واجهة جديدة يجب أن تستخدم نفس tokens والمكونات القائمة دون إنشاء أنماط متفرقة.

## 8.6 الوصولية والمرونة البصرية
1. نصوص عالية التباين فوق الخلفيات الزجاجية (لا يقل التباين عن الحد المقبول WCAG للقراءة).
2. دعم لوحة المفاتيح بالكامل مع `focus ring` واضح.
3. دعم `prefers-reduced-motion` لخفض الحركة.
4. دعم `prefers-reduced-transparency` مع fallback تلقائي عند غياب blur/transparency.
5. دعم responsive على شاشات اللابتوب والجوال بدون كسر RTL.

## 8.7 تخطيط لوحة التحكم (RTL Gov-Tech Layout)
1. التخطيط القياسي للوحة الرئيسية يجب أن يكون:
   - يمين: `Control Rail` (فلتر الفترة + الأوامر + الكونسول + قائمة التشغيلات).
   - يسار: `Selected Run Report` (KPI + الملاحظات + الشعب).
2. عرض عمود التحكم الأيمن بين `360px` و`420px` مع سلوك Sticky على الشاشات المكتبية.
3. أزرار العمليات يجب أن تتبع التسلسل:
   - أساسي: `تشغيل المعالجة`.
   - ثانوي: `تشغيل الفحوصات`.
   - Ghost: `تحديث التشغيلات` و`تحديث المحدد`.
4. يجب أن يوفر كونسول الاستجابة وظائف:
   - نسخ المحتوى.
   - طي/توسيع العرض.
   - تمرير داخلي دون توسيع البطاقة بالكامل.
5. يجب أن تستخدم الجداول `chips/badges` للحالة/الشدة وبعض الحقول المرجعية (مثل اليوم والقاعة) لتحسين القراءة السريعة.

1. يجب تطبيق ثيم:
   - `Charcoal + Emerald + Soft Silver`.
2. tokens الأساسية الإلزامية:
   - `--bg: #0B1220`
   - `--bg2: #070B14`
   - `--panel: rgba(255,255,255,0.08)`
   - `--panel2: rgba(255,255,255,0.06)`
   - `--panelBorder: rgba(255,255,255,0.12)`
   - `--text: #EAF0FF`
   - `--muted: #A9B4C7`
   - `--primary: #17C964`
   - `--primaryHover: #0EA75A`
   - `--accent: #D7DFEA`
   - `--danger: #FF4D4F`
   - `--warning: #F7B731`
   - `--info: #3B82F6`
3. يجب أن تكون الواجهة العربية RTL بخط `IBM Plex Sans Arabic` (أو fallback معتمد عند غيابه).
4. أي شاشة جديدة داخل المشروع يجب أن تلتزم بهذه الهوية دون إدخال palette ثالثة.

## 8.9 تعدد الثيم والكثافة (Theme & Density)
1. يجب دعم وضعين بصريين بنفس الهوية:
   - `dark`
   - `light`
2. التنفيذ يتم حصريًا عبر CSS variables:
   - `:root[data-theme=\"dark\"]`
   - `:root[data-theme=\"light\"]`
3. يجب حفظ تفضيل الثيم في `localStorage`.
4. عند عدم وجود تفضيل محفوظ، يعتمد النظام `prefers-color-scheme` كقيمة افتراضية.
5. يجب دعم وضعي كثافة:
   - `compact` (افتراضي)
   - `comfortable`
6. يجب حفظ تفضيل الكثافة في `localStorage` دون تعديل سلوك الأعمال.
7. تموضع بادج البيئة يكون داخل هيدر مخصص (`header-left-slot`) أعلى يسار الشاشة في جميع المقاسات.

## 8.10 شاشات الهِيت ماب متعددة الكيانات (Excel-like Multi Screens)
1. يجب أن توفر شاشة `التشغيل المحدد` أربع شاشات Heatmap منفصلة:
   - `القاعات`
   - `الشعب`
   - `المدربين`
   - `التوزيع النسبي`
2. كل شاشة يجب أن تعتمد مصفوفة زمنية موحدة:
   - 5 أيام (`الأحد` إلى `الخميس`)
   - 8 فترات (`1` إلى `8`)
3. يجب أن تدعم المصفوفات منطق القيم:
   - `0` = غير مشغول
   - `1` = مشغول
   - `>1` = تعارض/تكدس
4. يجب أن توجد فلاتر علوية موحدة تؤثر على جميع شاشات الهِيت ماب:
   - الفترة (`ALL`/`صباحي`/`مسائي`)
   - الإشغال (`all`/`occupied`/`empty`/`conflict`)
   - بحث نصي عن الكيان.
5. شاشة `التوزيع النسبي` يجب أن تعرض على الأقل:
   - مؤشر إشغال ثنائي
   - عدد الشعب في كل خانة
   - نسبة مئوية من إجمالي الشعب.
6. يجب أن تعرض كل شاشة ملخصًا علويًا سريعًا (عدد الصفوف/الخلايا المشغولة/التعارضات/نسبة إشغال).
7. تنفيذ هذه الشاشات يعتبر UI-only:
   - لا تعديل على backend business logic
   - لا تغيير على API contracts.

## 9. المتطلبات غير الوظيفية

### NFR-001 الأداء
1. زمن معالجة Run على حجم SS01 الحالي (حتى 5,000 صف): أقل من 30 ثانية.
2. زمن فتح جدول نتائج كبير: أقل من 2 ثانية للصفحة الأولى.
3. القياس يتم على بيانات مرجعية بحجم Beta6 ولمدة 10 تشغيلات متتالية.
4. معيار القبول للأداء يعتمد على `P95` وليس المتوسط فقط.
5. عند تجاوز حد 30 ثانية يجب أن يعمل التنفيذ كنمط غير متزامن مع مؤشر تقدم واضح.

### NFR-002 الاعتمادية
1. أي فشل في خطوة لا يفسد آخر نسخة ناجحة.
2. دعم إعادة تشغيل نفس Run deterministically.

### NFR-003 قابلية التدقيق
1. جميع التشغيلات قابلة للتتبع (Who/When/What).
2. حفظ نسخة input ونسخة output لكل Run.

### NFR-004 القابلية للصيانة
1. القواعد تحفظ كوظائف واضحة أو DSL داخل الكود، لا كصيغ خام.
2. كل قاعدة لها Test Cases.

### NFR-005 الأمان
1. RBAC وفق أدوار النظام.
2. سجل تدقيقي للعمليات الحساسة.
3. منع رفع ملفات غير مصرح بها.
4. سياسة احتفاظ: ملفات الإدخال الخام 180 يومًا، ومخرجات التشغيل والسجلات 365 يومًا (ما لم توجد سياسة مؤسسة أشد).
5. حقول مثل `رقم المدرب` تعامل كبيانات حساسة في التصدير والسجلات.
6. منع تنزيل المخرجات المنشورة إلا للأدوار المخولة.

### NFR-006 اللغة والاتجاه
1. العربية هي اللغة الأساسية.
2. RTL كامل في الواجهة والتقارير.

## 14. خطة الاختبار والتحقق

### 14.1 أنواع الاختبارات
1. Unit Tests لكل قاعدة حساب.
2. Integration Tests لكل Pipeline.
3. Snapshot Tests لمخرجات الجداول.
4. Regression Tests عند أي تغيير قواعد.

### 14.2 معايير المطابقة مع Excel
1. تطابق القيم الرقمية 100% (مع سماحية كسور إذا وجدت).
2. تطابق عدد الصفوف/الأعمدة في outputs الرئيسية.
3. تطابق مؤشرات الترويسة (المجاميع الأساسية).
4. تطابق حالات التلوين الحرجة (نقص/زيادة/تعارض).
5. قاعدة مقارنة الأرقام: `abs(system-excel) <= 0.000001` ما لم تنص القاعدة على غير ذلك.
6. قاعدة مقارنة الوقت: توحيد الصيغة إلى `HHMM` قبل المطابقة.
7. قاعدة المنطقة الزمنية: الحساب داخليًا بـ UTC والعرض حسب منطقة النظام المعتمدة.

### 14.3 حالات بيانات إلزامية
1. فترة صباحية كاملة.
2. فترة مسائية كاملة.
3. بيانات فيها تعارض ساعات واضح.
4. بيانات ناقصة (مدرب/قاعة غير معروفة).

## 19. قرارات تصميمية ملزمة

1. التنفيذ يكون داخل منصة SSP الحالية وليس نظامًا منفصلًا.
2. النموذج الحسابي يحول الصيغ إلى قواعد واضحة في الكود.
3. أي تغيير في قواعد الأعمال يتم عبر Versioned Rules.
4. المخرجات النهائية تحفظ كنسخ immutable مرتبطة بـ `run_id`.

## 20. القرارات المعتمدة قبل بدء التنفيذ

1. نطاق الاستيراد في الإصدار الأول: `SS01` فقط، وأي مصدر إضافي يدخل عبر Change Request موثق.
2. التصدير الرسمي الافتراضي: حزمة كاملة (`Halls + CRNs + Trainers + Distribution`) مع خيار `Halls` فقط عند الحاجة.
3. الصلاحيات المعتمدة: `Admin` (كامل)، `Operator` (استيراد/تشغيل/نشر/تصدير)، `Viewer` (قراءة فقط).
4. أي قاعدة تشغيل غير مثبتة في Beta6 تعتبر خارج النطاق حتى توثيقها كنسخة قواعد جديدة (`rule_version`).
5. أي تغيير على القرارات أعلاه يتطلب تحديث إصدار PRD قبل البدء في التطوير.
6. أي تعديل تنفيذي أو وظيفي يجب أن يحدّث معه `WORKLOG` و`HANDOVER` و`CHANGELOG` في نفس الدفعة.
